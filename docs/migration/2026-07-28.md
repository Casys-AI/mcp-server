# Migration Plan — MCP Specification 2026-07-28

Date: 2026-05-23   Status: Draft   Scope: monorepo (server, view, compose, bridge)

## Status

| Item | Value |
|---|---|
| Current supported spec | `2025-11-25` (stable) |
| Target spec | `2026-07-28` (Release Candidate, breaking) |
| Validation window | 2026-05-21 → 2026-07-28 |
| Final spec ship date | 2026-07-28 |
| Migration branch | _TBD — e.g. `spec/2026-07-28`_ |

This document is a plan, not a decision record. The decision to adopt
`2026-07-28` will follow a POC on the migration branch.

## Summary of Breaking Changes

The 2026-07-28 RC is the most invasive revision since the protocol's first
release. Key changes:

1. **Stateless core**. The `initialize` handshake and `Mcp-Session-Id`
   header are removed. Every request carries `protocolVersion` and
   `clientInfo` in `_meta`. Any request can land on any server instance.
2. **Multi Round-Trip Requests** replace persistent server→client SSE.
   A server may answer with an `InputRequiredResult` carrying request
   state; the client re-submits the request when ready.
3. **New routing headers**: `Mcp-Method` and `Mcp-Name` allow routing
   and rate-limiting without body inspection.
4. **Cache hints**: list/resource responses gain optional `ttlMs` and
   `cacheScope` (HTTP Cache-Control inspired).
5. **W3C Trace Context** is formally documented for distributed tracing.
6. **Extensions framework**: opt-in, independently versioned, reverse-DNS
   identifiers. Two official extensions launch alongside the spec:
   - **MCP Apps** — server-rendered HTML UIs in sandboxed iframes.
   - **Tasks** — redesigned for the stateless model.
7. **Authorization hardening** (six SEPs aligned with OAuth 2.0 / OIDC):
   `iss` validation per RFC 9207, `application_type` at DCR, refresh-token
   and scope-accumulation handling.
8. **Deprecations** (12-month removal window): **Roots**, **Sampling**,
   **Logging**.
9. **Error codes**: missing-resource error changes from `-32002` to the
   JSON-RPC standard `-32602`.

## Impact at a Glance

| Package | Impact | Notes |
|---|---|---|
| `@casys/mcp-server` | **High** | Transport, session model, push channel, deprecations |

## Server Migration — `@casys/mcp-server`

The plan is organised into five tracks. Tracks A, C, D, E are
independent and can run in parallel. Track B depends on A.

### Track A — Stateless Transport (effort: L)

Replace the stateful HTTP transport with a stateless one. The legacy
transport may live on temporarily, gated by an `enableStatelessV2`
option, to allow incremental migration of consumers.

**Affected code**:

| Location | Action |
|---|---|
| `packages/server/src/mcp-app.ts:1495–1542` | Drop `initialize`-driven session creation and the `sessions.has(reqSessionId)` validation block. |
| `packages/server/src/mcp-app.ts:1301` | Remove the `mcp-session-id` header read on GET/SSE routes. |
| `packages/server/src/mcp-app.ts:1507` | Update advertised `protocolVersion` to `2026-07-28` (negotiated, not hard-coded). |
| `packages/server/src/types.ts:445,577` and `packages/server/src/middleware/types.ts:25` | Remove `sessionId?: string` from `ToolContext`, `AppToolCall`, `HttpRateLimitContext`, `Middleware`. |
| `packages/server/src/runtime/` | Introduce a stateless POST handler that reads `_meta.protocolVersion` from the request body and dispatches to the existing tool/resource handlers. |

**Acceptance criteria**:

- A single `POST /mcp` request, with no prior handshake, succeeds when
  carrying a valid `_meta.protocolVersion`.
- No `Mcp-Session-Id` header is emitted or required.
- Round-robin across two instances behind a load balancer (no shared
  state) returns consistent results.

### Track B — Multi Round-Trip Requests (effort: L)

Replace the persistent server→client SSE channel with the
`InputRequiredResult` pattern. The client re-POSTs the request with a
continuation token when input becomes available.

**Affected code**:

| Location | Action |
|---|---|
| `packages/server/src/mcp-app.ts:232` (`sseClients` Map) | Decommission session-indexed SSE state. |
| `packages/server/src/mcp-app.ts:1297–1382` (GET `/mcp` SSE route) | Remove or feature-flag off; replace with a re-POST endpoint that resumes a pending operation. |
| `packages/server/src/mcp-app.ts:1901` (`sendToSession`), `1947` (`broadcastToAll`) | Re-architect around responses carrying `InputRequiredResult`; no more out-of-band push. |
| `packages/server/src/types.ts` | Introduce an `InputRequiredResult` type and integrate it in the tool-call response union. |

**Acceptance criteria**:

- A tool can return `InputRequiredResult` instead of a final result.
- The client can resume with a follow-up POST that lands on _any_
  instance and receives the next event.

### Track C — Routing Headers & Cache Hints (effort: M)

Decoration work; non-breaking on its own.

**Affected code**:

| Location | Action |
|---|---|
| `packages/server/src/mcp-app.ts` (HTTP response builders) | Emit `Mcp-Method` and `Mcp-Name` headers on responses. |
| `packages/server/src/mcp-app.ts` (`tools/list` and `resources/list` handlers) | Add opt-in `ttlMs` and `cacheScope` fields on list results, sourced from per-tool / per-resource registration metadata. |

**Acceptance criteria**:

- Headers appear on every response.
- A tool registered with a TTL surfaces `ttlMs` on `tools/list`.

### Track D — Deprecations (effort: S–M)

12-month removal window. Plan: warn now, remove in the release cut after
2027-07-28.

| Feature | Location | Action |
|---|---|---|
| Sampling | `packages/server/src/sampling/sampling-bridge.ts` + integrations at `packages/server/src/mcp-app.ts:208,284–285,1007–1009,1964–1967` | Emit a deprecation warning on `options.enableSampling = true`. Mark the public types `@deprecated`. |
| Logging (`logging/setLevel`) | `packages/server/src/mcp-app.ts:1689–1691` | Mark the stub `@deprecated` in code comments and release notes. |
| Roots | _not implemented_ | Nothing to do. Confirm absence in CI by a grep guard if desired. |

### Track E — Auth Hardening (effort: S)

| Item | Location | Status |
|---|---|---|
| `iss` validation (RFC 9207) | `packages/server/src/auth/jwt-provider.ts:251–253` | **Already covered** via `jose.jwtVerify(token, jwks, { issuer, audience })`. |
| CIMD client registration (SEP-991) | `packages/server/src/client-auth/` | **Implemented client-side, opt-in** via `clientRegistration.method = "client_id_metadata"` and `buildClientIdMetadataDocument()`. |
| DCR compatibility path | `packages/server/src/auth/as-metadata-proxy.ts` | Keep as legacy/interop support. Do **not** inject `application_type` here: it belongs to the DCR client-metadata payload, not to the AS metadata proxy. |
| Refresh-token / scope accumulation | `packages/server/src/client-auth/` | Audit token-store behaviour against the new SEPs once their text is final. |

### Track F — Error Codes (effort: S)

Change the missing-resource error code from `-32002` to `-32602`. Audit
all `JSON-RPC` error responses in `packages/server/src/` for similar
non-standard codes.

## View Migration — `@casys/mcp-view`

`@casys/mcp-view` is a thin wrapper over `@modelcontextprotocol/ext-apps`'
`App` class. The official MCP Apps extension (SEP-1865) ships with
`2026-07-28` and validates the package's positioning. No structural
migration is required, but two gaps are worth closing as the spec
stabilises:

1. **Missing helpers on `AppContext`** — the spec defines iframe→host
   requests such as `ui/request-display-mode`, `ui/update-model-context`,
   and `ui/open-link`. Today these are reachable only via the
   `ctx.app` escape hatch. Expose typed helpers: `ctx.requestDisplayMode`,
   `ctx.updateModelContext`, `ctx.openLink`.
2. **UI resource metadata** — the spec defines `_meta.ui.csp`,
   `_meta.ui.permissions`, and `_meta.ui.domain` on UI resources. These
   are server-side concerns (`@casys/mcp-server` resource registration)
   but should be modelled so that View consumers do not bypass them.
   Add a small `defineUiResource()` helper in `@casys/mcp-server` (or
   `@casys/mcp-compose`) that produces a spec-correct resource shape.

The internal router (`packages/view/src/router.ts`) remains a
differentiator: the MCP Apps spec does not define an intra-iframe
navigation primitive, so wrapping it inside View continues to add value.

## Compose & Bridge

No protocol-level changes are required. Compose and Bridge re-use the
server's transport. Once Track A lands, they pick up the stateless
transport for free.

## Validation Strategy

1. **Per-track POC** on a `spec/2026-07-28` branch. Each track lands as
   its own PR with tests.
2. **Independent review** of each PR via Codex (sub-agent reviewer) on
   architecture, safety, and conformance axes.
3. **Behind a feature flag** until the official `2026-07-28` spec ships,
   so `main` continues to publish a stable `2025-11-25`-compatible
   release.
4. **Interop matrix**: validate against at least one reference MCP host
   (Claude Code, MCP Inspector) on both the stateless transport and the
   legacy stateful path.

## Non-Goals

- Removing legacy Sampling / Logging integrations during the validation
  window. The 12-month deprecation clock starts at the final spec date;
  removal lands after 2027-07-28.
- Implementing the Tasks extension. Track B unblocks it but full Tasks
  support is out of scope until the extension API is final.
- Migrating downstream consumers automatically. The stateful transport
  remains available behind a flag for at least one minor release after
  `2026-07-28` ships.
