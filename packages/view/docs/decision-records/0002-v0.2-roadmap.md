# ADR 0002: `@casys/mcp-view` v0.2 Roadmap

Date: 2026-04-18  Status: Proposed

## Context

ADR 0001 records what v0.1.0 intentionally excluded. This ADR is the
complement: the ordered list of features likely to land in v0.2, with
a rationale for each so the scope stays explicit between releases.

Inputs:

- Items deferred in ADR 0001 (still valid).
- Patterns identified in `/home/ubuntu/CascadeProjects/mcp-einvoice/`
  (April 2026 scan) that are generic enough to promote, after critical
  filtering to remove design-locked, domain-specific, or framework-
  coupled code.
- Architecture decision from ADR 0001: vanilla-first, framework adapters
  go to separate packages (`@casys/mcp-view-react`, etc.).

## Candidates (ordered, highest priority first)

### 1. Auto-theme wire in `createMcpApp` — **0.1.1 patch candidate**

`@modelcontextprotocol/ext-apps` 1.6.0 ships `applyHostStyleVariables`,
`applyHostFonts`, `getDocumentTheme`, `applyDocumentTheme` (exported from
the main entry). At handshake the host sends `hostContext.styles` +
`hostContext.theme`; authors today would have to wire these calls
themselves.

**Proposal:** `createMcpApp` calls these helpers automatically after
successful `connect()`, before mounting `initialView`. Authors get
theme-applied-by-default with zero new public API.

**Scope:** ~15 lines in `app.ts`, 2 tests (theme applied / fonts applied).
No type surface change.

**Why 0.1.1 and not 0.2:** strictly additive, no API surface delta.
Safe to ship as a patch before v0.2 lands.

---

### 2. Refresh utils

Extract `canRequestUiRefresh`, `extractToolResultText`, and
`UiRefreshRequestData` from
`mcp-einvoice/packages/mcp/src/ui/shared/refresh.ts` (~63 lines). Pure
functions, zero framework coupling, zero einvoice domain knowledge. They
encapsulate visibility-aware throttling and the dual-shape tool result
payload extraction (`structuredContent` vs `content[0].text`) — both are
universal needs for MCP Apps that do background refresh.

**Proposal:** new sub-export `@casys/mcp-view/refresh` or folded into the
main `@casys/mcp-view` export. Leaning toward main export to avoid
paper-cut imports.

**Scope:** ~80 lines + tests. No runtime changes to `createMcpApp`.

---

### 3. viewerLifecycle vanilla core

`mcp-einvoice/packages/mcp/src/ui/shared/useViewerLifecycle.ts` (192
lines) is a React hook that wires `app.ontoolresult`,
`app.ontoolinputpartial`, throttles via (2), and auto-refreshes on
visibility/focus. The *hook* is framework-coupled; the *behaviour* is
universal.

**Proposal:** ship `createViewerLifecycle({ onRefresh, throttleMs, ... })`
as a non-hook function returning an object with `start()` / `stop()`.
Authors call it from `defineView.onEnter` / `onLeave`. A future
`@casys/mcp-view-react` can wrap this into `useViewerLifecycle`.

**Scope:** ~100 lines + tests. Depends on (2). API surface addition on
`AppContext` is discussed but not decided — the current lean is to keep
it out of `ctx` (separate callable) to avoid bloating the context object.

---

### 4. `ctx.sendMessage(text, { disclosure })` (from ADR 0001 §1)

Wrapper over `app.sendMessage` that **requires** an explicit
`disclosure` prop. The disclosure string gets rendered by the SDK before
the message is sent so the user sees exactly what will be injected into
the chat. Makes the SDK the discouragement mechanism for the anti-pattern
it was created to replace.

**Scope:** ~40 lines + integration test.

---

### 5. `ctx.requestDisplayMode(mode)` (from ADR 0001 §2)

Thin wrapper over `app.requestDisplayMode`, also exposes the current
mode via `ctx.displayMode` (read-only, updates on host-context-changed).

**Scope:** ~30 lines, depends on ext-apps' display-mode events being
listened to (part of handshake wiring — possibly same PR as #1).

---

### 6. `ctx.updateModelContext(payload)` (from ADR 0001 §3)

Wrapper — low priority. Only ships once we have a clear mental model
documented for authors: "when do I use callTool vs updateModelContext vs
sendMessage?" Documentation debt, not code debt.

---

### 7. Type-level view map inference (`keyof V`) (from ADR 0001 §6)

Requires `<S, V extends ViewMap<S>>` generics. Breaking change. Target
v1.0 to bundle with other breaking cleanups, not v0.2.

## Explicitly deferred further

Items from the einvoice scan that are **not** promoted — see the scan
transcript (`.remember/tmp/view-sdk-plan.md` neighbourhood for history):

- i18n framework: mechanics generic, strings specific; mature libs
  exist, not our business.
- Feedback/skeleton components: design-locked to Casys branding.
- Status lifecycle registry: einvoice domain-specific.
- `useViewerLifecycle` as a React hook: goes to `@casys/mcp-view-react`
  when that package ships, not here.

## Consequences

- Next view release is 0.1.1 (patch, item #1) if we ship auto-theme
  quickly. Otherwise skip straight to 0.2 with items 1–5 bundled.
- Item #5 may merge into #1 if we discover display-mode wiring touches
  the same lifecycle code.
- None of these items require breaking `AppConfig` / `ViewDefinition` /
  `AppContext` — ADR 0001's "all extension points are optional fields"
  promise stays intact.

## Trigger to promote candidates

A candidate upgrades from "Proposed" to "Accepted" when one of:

- einvoice (or another consumer) migrates to `@casys/mcp-view` and
  re-implements the pattern — concrete pull signal.
- A second MCP App hits the same pain in a review or issue.

Speculation alone doesn't promote — we shipped v0.1 by resisting scope
creep, keep that discipline.
